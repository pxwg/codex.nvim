OVERVIEW                                               *codex.nvim-overview*

codex.nvim runs `codex app-server --listen stdio://` and presents Codex
threads as Neovim buffers. A thread buffer is named:
>
  codex://thread/<thread-id>
<

The transcript is rendered above a `## Prompt` section. Type below that prompt
section and press CTRL-S to submit. Reasoning, tools, patches, plans, and agent
timeline items are rendered as extmark placeholder blocks. Unknown app-server
events are retained as raw event blocks and can be shown for debugging.

When a thread is opened from another buffer, codex.nvim tracks that source
buffer as the thread target for prompt context and Neovim dynamic tools.

File edit approvals are shown as patch proposals in a Neovim review window.
The `nvim.apply_patch` dynamic tool opens the edited file buffer directly for
hunk-by-hunk review.

SETUP                                                     *codex.nvim-setup*

Minimal setup:
>
  require("codex").setup()
<

Default options:
>
  require("codex").setup({
    app_server = {
      command = { "codex", "app-server", "--listen", "stdio://" },
      initialize_timeout_ms = 10000,
      sanitize_malloc_env = true,
    },
    thread = {
      model = nil,
      model_provider = nil,
      service_tier = nil,
      approval_policy = "on-request",
      approvals_reviewer = "user",
      sandbox = "workspace-write",
      permissions = nil,
      developer_instructions = nil,
      base_instructions = nil,
      personality = nil,
      ephemeral = false,
    },
    buffer = {
      on_attach = nil,
    },
    ui = {
      layout = "float",
      width = 0.82,
      height = 0.82,
      sidebar_width = 0.42,
      render_delay_ms = 35,
      auto_scroll = true,
    },
    render = {
      prompt_marker = "## Prompt",
      separator = "───",
      show_raw_events = false,
      virtual_blocks = {
        default_expanded = false,
        max_lines = 80,
        max_width = 180,
      },
    },
    completion = {
      enabled = true,
      ttl_ms = 30000,
    },
    edit = {
      mode = "pair", -- "pair" or "yolo"
    },
    dynamic_tools = {
      enabled = true,
      prefer_nvim_apply_patch = true,
    },
  })
<

On macOS, `app_server.sanitize_malloc_env` removes inherited
`MallocStackLogging*` variables before spawning app-server. This avoids noisy
malloc runtime messages from parent GUI environments; set it to `false` if you
intentionally need those variables while debugging Codex.

COMMANDS                                               *codex.nvim-commands*

                                                        *:Codex*
>
  :Codex new [initial prompt]
  :Codex open [thread-id]
  :Codex resume <thread-id>
  :Codex pick
  :Codex list
  :Codex submit
  :Codex stop
  :Codex detail
  :Codex health
  :Codex status
  :Codex restart
  :Codex attach [all]
  :Codex add-buffer
  :Codex add-selection
<

`pick` uses snacks.picker when available and falls back to |vim.ui.select()|.
`:Codex status` reports app-server state, active/current thread ids, pending
request counts, and the current thread generation/status.

`:Codex attach` reruns the configured Codex buffer attach hook for the current
thread buffer. `:Codex attach all` reruns it for every loaded Codex thread
buffer. Configure `buffer.on_attach = function(bufnr, payload) ... end` or
`require("codex").on("buffer_attached", cb)` to attach editor-local helpers
such as input-method LSP clients, formula concealers, or buffer-local keymaps.

Command-line completion covers subcommands, `attach all`, loaded Codex buffer
numbers, and loaded thread ids for `open`/`resume`.

`:Codex add-buffer` appends the current source buffer path to the active chat
prompt using direct `@path` syntax. `:Codex add-selection` appends
`@selection` when the source buffer has visual selection marks.

HEALTH                                                   *codex.nvim-health*

Run:
>
  :checkhealth codex
<

The health provider checks the Neovim version, `codex` executable,
app-server stdio support, app-server command shape, the Codex apply_patch
runtime for `nvim.apply_patch` in pair mode, optional picker and completion
integrations, and dynamic tool registration. `:Codex health` performs the
runtime app-server initialization check.

PROMPT TOKENS                                    *codex.nvim-prompt-tokens*

The blink source and parser recognize these prompt token families:
>
  $skill:<name>

  /model
  /permissions
  /mcp
  /status

  @file:
  @image:
  @buffer
  @selection
  @cursor
  @diagnostics
  @quickfix
  @buffers
  @cwd
  @path/to/file
<

`$skill:<name>` candidates come from Codex app-server `skills/list`. `/...`
candidates are CLI-style slash commands such as `/model`, `/permissions`,
`/mcp`, and `/status`. Slash commands are handled by codex.nvim before
`turn/start`, so they are not sent to the model as tool-call-like prompt text.
Accepting a slash completion removes the typed prefix and opens that command's
page or picker instead of inserting text. Each slash command declares a return
form (`page`, `select`, `notify`, `insert`, or `action`) and uses one presenter
for Neovim rendering. Settings commands such as `/model`, `/fast`,
`/permissions`, `/sandbox`, `/reasoning`, `/personality`, and `/experimental`
open Neovim pickers backed by Codex app-server catalog responses where
available and update the active thread where app-server supports it. `/model`
also offers the selected model's advertised thinking-effort choices when
app-server returns them.

`@...` candidates are Neovim context shortcuts. They expand into additional
Codex inputs before the turn is sent. Argument providers use
`@provider:input`; paths with spaces can wrap the path in backticks:
>
  @file:`path with spaces.lua`
  @image:`assets/screenshot.png`
  @lua/codex/init.lua
<
The blink source completes paths after `@file:` and `@image:` using that
backtick form. In insert mode, pressing TAB immediately after `@file:` opens
`snacks.picker.files` when available, with |vim.ui.select()| only as a
fallback; the selected file is inserted as direct `@path/to/file` syntax.
`@image:` keeps the provider form because image inputs need image-specific
attachment metadata. Custom hooks can be registered with
`require("codex.context").register_hook(name, callback)`.

When a thread is opened from another window, codex.nvim remembers that source
buffer as the thread target, so `@buffer`, `@selection`, `@cursor`,
`@diagnostics`, and Neovim dynamic tools do not read the chat buffer itself.
`@selection` reads the source buffer's visual selection marks and includes file
and range metadata plus diagnostics in the selected range. If the source buffer
has a visual selection, that selection context is automatically attached to
each submitted prompt unless the prompt already contains `@selection`.
Expanded text contexts are sent before the user request and are labeled as
reference context, not instructions; the user request remains the final text
input for semantic priority.
`@buffer` includes buffer RPC metadata such as bufnr, path, filetype, line
count, cursor, modified state, and buffer text.
Legacy `>buffer`, `>diagnostics`, and `>quickfix` still parse as context
aliases, but new completions use `@`.

PATCH REVIEW                                      *codex.nvim-patch-review*

Codex app-server file-change approvals are normalized into a patch proposal.
The review window supports:
>
  a     accept
  A     accept for session
  d     decline
  c     cancel
  [c    jump to previous file change or diff hunk
  ]c    jump to next file change or diff hunk
  <CR>  open related file at the hunk location when available
  o     open related file at the hunk location when available
  q     close window without answering
<

The review buffer indexes file changes and unified-diff hunk headers with
extmarks so larger patches can be navigated as editor state.

Modern app-server requests use `item/fileChange/requestApproval`. Legacy
`applyPatchApproval` requests are supported as a fallback.

The `nvim.apply_patch` dynamic tool uses an in-buffer flow instead. It accepts
Codex's native `apply_patch` patch format, verifies it against a temporary copy
of the edited files, converts the result to review hunks, opens the edited file
directly, previews proposed changes as real buffer edits with old content shown
as virtual lines, and writes only after hunks are resolved. Legacy unified diffs
are still accepted when `git` is available.

`edit.mode = "pair"` is the default. In pair mode, codex.nvim exposes
`nvim.apply_patch` and adds harness instructions requiring Codex's native
`apply_patch` format with small, focused patches, preferably one logical edit
per call. The agent is told to read the current buffer or relevant current file
content before patching, build the patch from exact current content, use
relative file paths, and re-read the buffer before retrying any failed patch. It
must not call native `apply_patch` directly unless `nvim.apply_patch` reports
that native fallback was approved for the current turn.

`edit.mode = "yolo"` disables the `nvim.apply_patch` dynamic tool and tells
Codex to use the native `apply_patch` tool directly for workspace edits. Calls
to `nvim.apply_patch` are rejected while yolo mode is active, so the review
tool is only available in pair mode. The legacy option
`dynamic_tools.prefer_nvim_apply_patch = false` selects yolo mode unless
`edit.mode` is set explicitly.

In a `nvim.apply_patch` buffer review:
>
  <leader>ca  accept current hunk
  <leader>cr  reject current hunk and prompt for a reason
  <leader>cA  accept all remaining hunks
  <leader>cR  reject all remaining hunks and prompt for a reason
  <leader>cf  use native apply_patch for the current turn
  <leader>cq  cancel the review
  [c          previous pending hunk
  ]c          next pending hunk
<
Rejected hunk reasons, partial-apply status, final file state, a final diff, and
the target buffer's `nvim.diagnostics` output are returned to Codex as the
dynamic tool result so the agent can continue from the user's feedback. In pair
mode, codex.nvim preserves user-provided developer instructions and appends the
edit protocol as additional thread developer instructions.

TUI RENDERING                                          *codex.nvim-tui*

The renderer follows a modern Neovim TUI shape:
>
  Codex ThreadItem -> Codex block -> rendered buffer lines -> extmark UI
<

The block types include:
>
  UserBlock
  AssistantBlock
  ReasoningBlock
  PlanBlock
  ToolCallBlock
  PatchBlock
  AgentTimelineBlock
  RawEventBlock
<

Headers are rendered through overlay extmarks. Reasoning/tool/patch/agent
blocks are placeholders with virtual body lines. Busy turns show a spinner
whose label follows the current generation state: thinking, reasoning,
streaming, tooling, patch review, syncing, or stopping.

The renderer tracks per-window view state. A window near the prompt follows the
composer while Codex streams. Cursor movement or scrolling away suspends that
follow state for that window. The prompt line is restored by anchor instead of
raw line number so streamed output does not shove the composer out of view.

Rendered user/assistant/error sections populate Vim fold ranges through
`v:lua.CodexFoldExpr(v:lnum)`. Codex windows keep `foldlevel` high by default,
so the folds are available without collapsing the transcript on open.

App-server lifecycle notifications such as hooks, auto-approval reviews,
server-request resolution, context compaction, and model reroutes become
timeline blocks. Set `render.show_raw_events = true` to also show retained raw
notification blocks.

Thread buffer keys:
>
  CTRL-S    submit prompt
  za        expand/collapse placeholder block
  K         open full block detail
  q         close window
<

EVENTS                                                  *codex.nvim-events*

codex.nvim emits User autocmds for editor integrations:
>
  CodexBufferAttached
<
    Fired after the Codex buffer attach hook point runs for a thread buffer.
    `event.data` includes `bufnr`, `thread_id`, and `thread`.
>
  CodexBufferOpened
<
    Fired after a thread buffer is opened in a window. `event.data` includes
    `bufnr`, `winid`, `thread_id`, and `thread`.
>
  CodexThreadOpened
<
    Fired when app-server reports a thread start.
>
  CodexGenerationCompleted
<
    Fired when app-server reports a completed generation.

Legacy snake_case User patterns are still emitted for compatibility.

LUA API                                                  *codex.nvim-api*

                                                        *codex.setup()*
setup({opts})~
    Configure codex.nvim.

                                                        *codex.new_thread()*
new_thread({opts})~
    Start a Codex thread and open its buffer.

                                                        *codex.open()*
open({thread_id})~
    Open a loaded thread or read a thread by id.

                                                        *codex.resume()*
resume({thread_id})~
    Resume a thread by id.

                                                        *codex.submit()*
submit()~
    Submit the current Codex buffer prompt.

                                                        *codex.submit_text()*
submit_text({text}, {thread_id})~
    Submit text to a thread.

                                                        *codex.stop()*
stop()~
    Interrupt the active turn.

                                                        *codex.detail*
:Codex detail~
    Open a detail buffer for the block under cursor.

                                                        *codex.list_threads()*
list_threads({callback})~
    List threads for the current workspace.

                                                        *codex.pick_thread()*
pick_thread()~
    Pick and resume a thread.

                                                        *codex.health()*
health()~
    Start app-server if necessary and report whether it initialized.

                                                        *codex.status()*
status()~
    Return lightweight runtime state for statuslines and integrations.

                                                        *codex.attach_buffer()*
attach_buffer({bufnr})~
    Rerun `buffer.on_attach` and `buffer_attached` hooks for a Codex buffer.

                                                        *codex.attach_all_buffers()*
attach_all_buffers()~
    Rerun Codex buffer attach hooks for every loaded Codex thread buffer.

                                                        *codex.add_current_buffer()*
add_current_buffer()~
    Append the current source buffer path to the active chat prompt.

                                                        *codex.add_selection()*
add_selection()~
    Append `@selection` to the active chat prompt when the source buffer has a
    visual selection.

                                                        *codex.on()*
on({event}, {callback})~
    Register a Lua hook. Returns a function that removes the listener.
